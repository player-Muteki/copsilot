# Copsidian 项目规划与执行报告

## 一、 现状与架构分析

Copsidian 是一个成熟的 Obsidian 插件，将 OpenCode Agent 无缝嵌入到 Obsidian 侧边栏中。目前架构清晰，模块化程度较高：

*   **客户端层 (`src/client`)**：通过 ACP (Agent Client Protocol) 使用 stdio/JSON-RPC 协议与本地 OpenCode CLI 通信，管理进程生命周期和异常处理。
*   **视图层 (`src/view`)**：负责侧边栏界面 (`copsidianView.ts`)、消息渲染、行内编辑、拖拽以及配置信息的展示。
*   **聊天逻辑 (`src/chat`)**：管理会话状态、多会话存储（支持持久化）和流式响应控制器。
*   **上下文管理 (`src/context`)**：实现对 Obsidian 笔记的 `@提及`、解析和提示词注入。
*   **同步引擎 (`src/sync`)**：捕获 Agent 的工具调用，并通过模板自动将修改或新建的内容同步回 Vault 笔记。

**当前瓶颈与改进空间**：
1. **测试覆盖率分布不均**：核心同步和客户端协议（ACP）的测试较好，但视图层 (`src/view`) 和聊天层 (`src/chat`) 覆盖率偏低（大部分在 20%~50%），UI 与交互状态容易在后续迭代中发生回归问题。
2. **缺乏压力测试**：系统缺乏在极端数据（超长上下文、超多并发会话）和异常状态（CLI 突然崩溃重启）下的系统级测试保障。
3. **高级功能扩展**：现有 Roadmap 基础功能已实现，下一步需要往深度（知识管理增强）和易用性（导出、快捷指令）发展。

## 二、 路线图与新功能落地规划

为了进一步完善核心体验，拟在下一阶段落地以下新特性：

### 1. 会话记录的导出与分享
*   **目标**：允许用户将当前优质的 AI 会话上下文导出为标准的 Obsidian Markdown 文件，便于归档和复用。
*   **实现方案**：
    *   在视图层 Toolbar 中增加“导出为笔记”的按钮。
    *   解析 `SerializedMessage` 数组，将其格式化为易读的 Markdown 文本（含 System Prompt 和 Assistant 回复）。
    *   调用 Obsidian API，在指定（或默认）目录生成文档。

### 2. 轻量级知识库检索（轻量级 RAG）
*   **目标**：当用户在侧边栏询问宽泛问题时，Agent 能够自动检索当前 Vault 中相关的笔记并作为上下文回答，而不需要手动 `@提及` 每一篇。
*   **实现方案**：
    *   增加一个全新的 Tool（例如 `search_vault`），允许 Agent 发起文本搜索。
    *   后端利用 Obsidian 的 `app.metadataCache` 和正则/关键字匹配，快速检索相关文本片段，并将结果注回 Agent。

### 3. 更多快捷命令 (Slash Commands)
*   **目标**：增强聊天输入框的交互效率。
*   **实现方案**：
    *   新增 `/clear`（清空当前会话上下文）。
    *   新增 `/export`（触发会话导出）。
    *   新增 `/system`（临时修改当前对话的 System Prompt）。

## 三、 测试覆盖率提升计划

目前 `src/view` 和 `src/chat` 测试覆盖率较低，目标是在下一阶段将这部分核心业务逻辑的覆盖率提升至 **85% 以上**。

### 1. `src/chat` 模块 (聊天与流式逻辑)
*   **`streamController.ts` (当前 13%)**：重点补充对事件流的分块解析、各种更新指令（如 `sessionUpdate` 类型的多种细分消息）的状态管理测试。
*   **`input.ts` (当前 29%)**：补充用户输入框事件处理（键盘事件拦截、文本注入）的用例。

### 2. `src/view` 模块 (界面与渲染逻辑)
*   **`copsidianView.ts` (当前 32%)**：通过 Mock Obsidian 的 API（WorkspaceLeaf），测试视图的初始化、生命周期（onload/onunload）以及与 sessionStore 之间的数据绑定逻辑。
*   **`renderer.ts` (当前 24%)**：补充对于不同消息类型的 Markdown 渲染逻辑，包含 Tool Call 的高亮和 Diff 渲染测试。
*   **`sessionDropdown.ts` (当前 11%) / `autocomplete.ts` (当前 5%)**：补充列表项的焦点切换、回车选中回调触发的单元测试。

## 四、 全场景压力测试方案

为了保障插件的高可用性和可靠性，必须引入一套全面的压测体系。

### 1. 高并发消息与操作压测
*   **场景**：同时快速触发多次 Tool Call，或者短时间内向 Agent 发送 100+ 条短消息。
*   **验收标准**：消息队列不阻塞，前端渲染无明显卡顿或内存泄露，ACP 层的读写流不会出现破损或丢包。

### 2. 大数据量上下文压测
*   **场景**：通过 `@提及` 引入单篇超过 500KB 的大笔记，或者同时引入数十篇笔记进入上下文。
*   **验收标准**：能正确触发 `maxNoteSize` 的截断逻辑；UI 层在加载极大量文本时不会导致主线程阻塞。

### 3. 并发会话与同步压力
*   **场景**：同时打开和维护 50+ 个会话（在 `PluginData` 中），并频繁在不同会话中触发 `SyncEngine` 写入 Vault。
*   **验收标准**：`Mutex` 锁（如 `sync/engine.ts` 和 `view/copsidianView.ts` 中的）能有效防止文件读写冲突；Obsidian 不抛出“文件被占用”异常。

### 4. 极端故障与断连恢复压测
*   **场景**：在消息生成过程中，强制 kill 掉本地的 `opencode` CLI 进程。
*   **验收标准**：插件需能正确捕获断连异常（通过 `AcpSubprocess`），在 UI 上给出恰当提示，并支持通过操作自动拉起重连机制，而不会导致整个 Obsidian 闪退或进入无响应状态。

## 五、 下一步详细执行计划（Phase 1 - Phase 3）

### 第一阶段：测试基建与覆盖率补全 (预计周期: 1 周)
1. **任务**：针对 `src/chat` 编写单元测试，完成 `streamController` 与 `input` 的 85% 覆盖率。
2. **任务**：重构并增加对 `src/view` 的 Mock 测试支持，补充 `copsidianView` 和各类 UI 组件的单元测试。
3. **结果**：整体代码库覆盖率指标通过 Vitest 校验达到预期。

### 第二阶段：全场景压力测试构建 (预计周期: 1 周)
1. **任务**：编写专门的压测脚本/套件，模拟大量的消息输入和大型 Context 注入。
2. **任务**：实现容灾机制测试用例，模拟 CLI 崩溃场景下的自动恢复/提示。
3. **结果**：形成稳定的 CI 压力测试流，输出压力测试性能报告。

### 第三阶段：功能落地与迭代 (预计周期: 2 周)
1. **任务**：实现“会话导出为笔记”功能及 `/export` 命令。
2. **任务**：开发 `search_vault` 工具集，接入简单的检索逻辑（轻量 RAG 雏形）。
3. **任务**：更新使用手册与 README，并发布新的 Release 版本。
4. **结果**：完成全部路线图扩展功能的合并。
